SQL Server / PostgreSQL Case Study #3

Composite Index vs. Full Table Scan

The problem: A query against a table with 2.4 million rows filtered on `Status` and `CreatedAt`. With no matching index in place, the database had to scan the entire table on every call (full table scan) to find matching results. This saturated the CPU and took nearly 5 seconds.

Baseline (no index / table scan)
✕ Execution plan: clustered index scan (2.4M reads, 4,800 ms)
SELECT Id, CustomerNumber, Amount 
FROM Orders 
WHERE Status = 'Active' AND CreatedAt >= '2026-01-01';
The optimization (composite index)
✓ Index definition for pinpoint data access
-- Creates a multi-column index that exactly covers the WHERE clause
CREATE INDEX IX_Orders_Status_CreatedAt 
ON Orders (Status, CreatedAt) 
INCLUDE (CustomerNumber, Amount);

The solution: By creating a composite index, the database jumps straight to the matching rows via a B-tree (index seek). The `INCLUDE` clause keeps the remaining columns directly in the index, eliminating any access to the base table entirely (covering index).

Performance gain: 2400× faster (down to 2 ms)
