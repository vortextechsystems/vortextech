Unity / C# / DOTS Case Study #4 — In Development

Unity ECS vs. MonoBehaviour Performance

The context: For our own Steam game, we needed to decide how to manage thousands of simultaneously active game objects (enemies, projectiles, particle interactions). Classic object-oriented MonoBehaviour design scales poorly at high entity counts, since every instance carries individual memory overhead and causes cache misses.

Baseline (MonoBehaviour)
✕ Object-oriented: every entity is its own GameObject with its own memory block
public class Enemy : MonoBehaviour
{
    public float speed = 3f;
    public Transform target;

    void Update()
    {
        // Per frame & per object: individual method call,
        // poor cache locality at thousands of instances
        transform.position = Vector3.MoveTowards(
            transform.position, target.position, speed * Time.deltaTime);
    }
}
Optimization (ECS + Burst Compiler)
✓ Data-oriented: compact arrays, parallelizable, Burst-compiled
[BurstCompile]
public partial struct EnemyMoveJob : IJobEntity
{
    public float DeltaTime;

    void Execute(ref LocalTransform transform, in MoveSpeed speed, in TargetPosition target)
    {
        // Runs in parallel over compact, cache-friendly data arrays
        // instead of scattered object instances
        transform.Position = Vector3.MoveTowards(
            transform.Position, target.Value, speed.Value * DeltaTime);
    }
}

The takeaway: By switching to Unity's Entity Component System (DOTS) with Burst-compiled jobs, movement calculations are distributed in parallel across multiple CPU cores and benefit from significantly better cache locality. The result is an architecture that holds stable frametimes even as object counts scale sharply — direct evidence of the same performance mindset we already apply to backend and database optimization.
