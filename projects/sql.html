SQL Server / PostgreSQL Case Study #3

Composite Index gegen Full Table Scan

Das Problem: Eine Abfrage auf eine Tabelle mit 2,4 Millionen Zeilen filterte nach `Status` und `ErstelltAm`. Da kein passender Index existierte, musste die Datenbank bei jedem Aufruf die komplette Festplatte scannen (Full Table Scan), um die passenden Ergebnisse zu finden. Das blockierte die CPU und dauerte fast 5 Sekunden.

Ausgangslage (Kein Index / Table Scan)
✕ Execution Plan: Clustered Index Scan (2,4 Mio. Reads, 4.800 ms)
SELECT Id, KundenNummer, Betrag 
FROM Bestellungen 
WHERE Status = 'Aktiv' AND ErstelltAm >= '2026-01-01';
Die Optimierung (Verbundindex)
✓ Index-Definition für punktgenauen Datenzugriff
-- Erstellt einen mehrspaltigen Index, der die WHERE-Klausel exakt abdeckt
CREATE INDEX IX_Bestellungen_Status_ErstelltAm 
ON Bestellungen (Status, ErstelltAm) 
INCLUDE (KundenNummer, Betrag);

Die Lösung: Durch die Erstellung eines Composite Index (Verbundindex) springt die Datenbank über einen B-Tree direkt zu den gesuchten Zeilen (Index Seek). Das Schlüsselwort `INCLUDE` sorgt dafür, dass auch die restlichen Spalten direkt im Index liegen. Ein Zugriff auf die Haupttabelle entfällt komplett (Covering Index).

Performance-Gewinn: 2400× schneller (nur noch 2 ms)
