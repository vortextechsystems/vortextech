Unity DOTS / Burst Compiler Case Study #4

Unity ECS vs. MonoBehaviour Performance

Das Problem: Klassische, objektorientierte GameObjects (`MonoBehaviour`) verursachen massiven Overhead durch Heap-Allokationen und verstreute Speicheradressen im RAM (Cache Misses). Ab ca. 5.000 aktiven Einheiten auf dem Bildschirm brechen die Frametimes drastisch ein.

Klassischer Ansatz (Objektorientiert)
✕ Jedes Objekt hält eigene Daten und Logik (Verstreut im RAM)
public class Enemy : MonoBehaviour 
{
    public float speed = 5f;
    void Update() {
        // Verursacht zehntausende CPU-Umsprünge pro Frame
        transform.Translate(Vector3.forward * speed * Time.deltaTime); 
    }
}
Datenorientierter Ansatz (Entity Component System)
✓ Reine Datenstrukturen, parallelisiert über alle CPU-Kerne
// 1. Reine Datenkomponente (Blittable Struct, kompakt im RAM arrangiert)
public struct MoveSpeed : IComponentData { public float Value; }

// 2. Isoliertes System (Nutzt den Burst Compiler und das Job-System)
[BurstCompile]
public partial struct MoveSystem : ISystem 
{
    [BurstCompile]
    public void OnUpdate(ref SystemState state) {
        // Verarbeitet Millionen Transformationen direkt im schnellen CPU-Cache
        new MoveJob().ScheduleParallel(); 
    }
}

Das Ziel: Umstellung auf das datenorientierte Technologie-Stack (DOTS) von Unity. Daten werden lückenlos in Arrays hinterlegt, sodass die CPU sie via SIMD-Instruktionen hocheffizient abarbeiten kann. Dies bildet das Fundament für die anspruchsvolle Performance unseres eigenen Steam-Titels.
