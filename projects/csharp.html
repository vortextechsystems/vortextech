ASP.NET Core / Async Case Study #1

Async Deadlock durch .Result / .Wait()

Das Problem: In einer Legacy-Anwendung blockierten synchrone Aufrufe von asynchronen Methoden (Sync-over-Async) den Thread-Pool. Unter hoher Last führte dies zu einem Deadlock des Synchronisationskontexts, wodurch alle nachfolgenden HTTP-Anfragen einfroren.

Fehlerhafter Code (Blockierend)
✕ Anti-Pattern: Erzwingt synchrone Blockierung
public IActionResult GetData()
{
    // .Result blockiert den Thread, bis die Task abgeschlossen ist
    var data = FetchDataAsync().Result; 
    return Ok(data);
}
Optimierter Code (Asynchron)
✓ Best Practice: Non-blocking via async/await
public async Task<IActionResult> GetDataAsync()
{
    // Gibt den Thread während des I/O-Wartens an den Pool frei
    var data = await FetchDataAsync(); 
    return Ok(data);
}

Die Lösung: Die gesamte Aufrufkette wurde konsequent auf native Asynchronität umgestellt (`async/await`). Dadurch blockieren wartende Threads keine wertvollen Systemressourcen mehr.

P99-Latenz nach Fix: < 120 ms
Entity Framework Core Case Study #2

N+1-Query-Problem mit EF Core

Das Problem: Durch aktiviertes Lazy Loading generierte eine Abfrage für 1.000 Datensätze unbemerkt 1.001 separate SQL-Statements, da verknüpfte Entitäten innerhalb einer Schleife einzeln nachgeladen wurden. Die Datenbankverbindung wurde komplett überlastet.

Fehlerhafter Code (Lazy Loading)
✕ Generiert 1.001 Queries (Laufzeit: 4,2s)
// Lädt nur Kunden, verknüpfte Bestellungen werden in der Schleife einzeln abgefragt
var customers = _context.Customers.ToList(); 
foreach(var c in customers) {
    Console.WriteLine(c.Orders.Count); 
}
Optimierter Code (Eager Loading)
✓ Generiert genau 1 Query via JOIN (Laufzeit: 38ms)
// Lädt Daten vorab via SQL-JOIN und deaktiviert das EF-Change-Tracking für maximalen Speed
var customers = _context.Customers
    .Include(c => c.Orders)
    .AsNoTracking()
    .ToList();

Die Lösung: Umstellung auf Eager Loading mittels `.Include()`. Das zusätzliche `.AsNoTracking()` spart CPU-Zyklen und Arbeitsspeicher, da die Daten im Lesezugriff nicht vom Entity-Framework-Kontext überwacht werden müssen.

Performance-Gewinn: 110× schneller
